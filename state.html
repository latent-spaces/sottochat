Architecture

Core loop: tailer → turns → observer → chat.

  • src/tailer.ts — polls ~/.claude/projects/* + codex rollouts; parses each jsonl line; emits MetaEvents. Now also lifts the raw cwd onto SessionInfo.cwd.
  • src/turns.ts — closes a turn on a stop event or a new user message.
  • src/observer.ts — one long-lived claude-sonnet-5 subprocess, batched every 30s (META_OBSERVER_BATCH_MS). Now a summarizer: returns one {turnId, summary} per fed session; the summary is "what the session is about", in the user's language.
  • src/chat-agent.ts — one claude-sonnet-5 subprocess per session, tools disabled, text-in/out. Answers in the chosen language; emits a fenced to-agent block when asked to draft a reply. send() takes {seed, language}; persona set on first message; language re-asserted on change; respawns clean on stop().
  • src/server.ts — Bun HTTP + ws on 3737. Holds SessionState (events, turns, summary/summaryTs, closedTurnCount, recentClosedTurns, contextTokens, totalOutputTokens) and the global explainLang. Feeds the summarizer on the 1st fresh closed turn then every 4th.
  • public/index.html — single-file SPA. Sidebar cards (summary + cwd + token chip), detail pane (header + charts + conversation strip + chat thread + ask box), language pill, copy-card. marked + gsap via CDN.
  • Other src: jsonl, registry, claude-discovery, codex-discovery, codex-jsonl, process-discovery, triggers.

Env knobs: META_PORT (3737), META_OBSERVER_MODEL, META_CHAT_MODEL, META_EXPLAIN_LANG (he), META_OBSERVER_BATCH_MS, META_OBSERVER_ENABLED=0 to skip the summarizer subprocess.

Decisions (why)

  • Observer → summarizer. The per-turn insight + editable prefill mechanic was removed. The card now shows a one-sentence "what this session is about", regenerated on the 1st fresh turn then every 4 (a throttle, not per-turn). "About", not activity narration — the owner corrected this explicitly.
  • No write channel to the terminal. The app only reads jsonl. "Reply to the agent" can only mean a copy-paste block: the assistant wraps the reply in a fenced to-agent block → the frontend renders a one-click copy card (LTR, agent's language). This is load-bearing.
  • Language is a global setting: curated dropdown (he default + en/ar/es/fr/ru/de/zh), localStorage + POST /settings/language + ws settings:language. Drives everything said to the user (summaries + answers); the drafted reply stays in the agent's own language.
  • Working dir comes from the raw jsonl cwd field (top-level on user/assistant records) — never derived from the slug, which mangles dashes (claude-meta → claude/meta).
  • Token count shown = contextTokens (current context-window size), not total output — it's the "how big is this session" number PRODUCT.md already names as the audience's vocabulary.
  • Model = claude-sonnet-5 for both the summarizer and the chat assistant (env-overridable).
  • Clear = drop + forget. "Clear discussion" (owner chose over archive) stops the chat subprocess (assistant forgets) and wipes the thread; the next send respawns fresh, re-seeded with the latest exchange.

Gotchas — read before touching

1. public/index.html contains literal 0x01 bytes

Some JS string literals use a SOH (\x01) control byte as a separator (e.g. .join("\x01") in cardSig, and the " " separators in sortedSessions). Plain grep treats the whole file as binary and silently returns nothing. Use grep -a or awk 'index($0,pat){print NR}' to search. The Edit tool will fail to match any multi-line old_string spanning those lines — edit only the clean lines around them.

2. Port 3737 is the owner's live app (old code)

bun src/server.ts on 3737 fails "port in use" but silently — then any browser/curl hits the old running app, so your changes look like they regressed (this cost real debug time; cwd looked "broken", it was the old build). Always test on a fresh port (META_PORT=3943+) and confirm via the boot log line listening on http://localhost:PORT.

3. chrome-devtools MCP browser wedged

After a test server died mid-session, the devtools browser stuck on "browser already running for …/chrome-profile" and every navigate/screenshot failed. Recovery not established this session — likely needs the chrome process killed, or the --isolated flag. DOM inspection via evaluate_script worked before it wedged and is the reliable way to verify frontend state (no screenshot needed).

4. Summaries only fire on fresh turn closes

The cadence counts fresh turns, not the startup backfill (a review fix). To see a summary in a test you need a live session to actually close a turn — use META_OBSERVER_BATCH_MS=6000 and poll /state for .summary. Each observer/chat call spawns a real claude subprocess (needs claude on PATH + auth) and takes ~5–30s.

Contracts

HTTP routes

  • GET /, /state, /sessions, /diag/discovery*
  • POST /chat/send {sessionKey,text} — sends to the chat assistant (seeded with cwd + summary + latest exchange, in explainLang)
  • POST /chat/clear {sessionKey} — drops subprocess + wipes thread + broadcasts chat:cleared
  • POST /settings/language {language} — validated via hasOwnProperty; no-op guarded; broadcasts on change

ws message kinds

hello{sessions,language} · session:upsert · event · thread:new · session:summary{sessionKey,summary,summaryTs} · chat:chunk · chat:status · chat:cleared{sessionKey} · settings:language{language}

The to-agent marker

The chat assistant wraps a suggested reply in a fenced block tagged to-agent. The frontend peels it into a copy card with a line-anchored regex (so an inline mention in the explanation can't false-match):

/(?:^|\n)[ \t]*```to-agent[^\n]*\n([\s\S]*?)\n[ \t]*```/
